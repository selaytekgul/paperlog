# Paperlog handoff — 2026-07-12

This document records the state of Paperlog at the end of the first implementation and launch day. It intentionally contains no passwords, access tokens, API keys, private database exports, or tester email addresses.

## Current outcome

- Product: a social reading diary for published and preprint research papers.
- Primary URL: `https://paperlog.net`
- Alternate URL: `https://www.paperlog.net`
- Current hosting: OpenAI ChatGPT Sites, public-beta hosting.
- Domain registrar and DNS control: METUnic; the domain remains owned independently of the hosting provider.
- Source location: this repository; Git is the source of truth.
- Current audience: public reading with authenticated writing, intended for a controlled adult friend alpha before wider promotion.
- Current database: a Cloudflare D1-compatible SQLite database supplied through the hosting environment as the `DB` binding.
- Scholarly metadata: OpenAlex search and lookup, plus direct DOI, arXiv, OpenReview, and OpenAlex identifier handling.

The Sites project identifier is stored in `.openai/hosting.json`. It is deployment metadata, not a credential. Secrets must remain in the hosting environment and in ignored local `.env` files.

## Product decisions

- Paperlog is closer to Letterboxd or Goodreads for papers than to formal peer review.
- Short, personal, early, or imperfect reader reactions are welcome when clearly presented as reader notes.
- The first rating model is a simple one-to-five-star score plus an optional comment.
- Reading status gives context: want to read, reading, read, skimmed, or worked through.
- Reproducibility is represented separately through structured code-experience reports, including repository, commit, environment, data, outcome, score, notes, and evidence.
- Paper profiles are created incrementally. OpenAlex provides discovery and metadata; a paper is cached locally when it becomes relevant to Paperlog activity.
- Paperlog does not host publishers' PDFs. It stores bibliographic metadata, user-generated notes, ratings, and links.

## Important application surfaces

- `/` — landing page, search, featured papers, and recent community activity.
- `/explore` — search and discovery results.
- `/paper/:id` — a paper profile with logs, rating summary, and reproducibility experiences.
- `/profile` — the signed-in reader's profile and account actions.
- `/reader/:slug` — a public reader profile.
- `/alpha` — the alpha testing instructions.
- `/admin` — moderation, corrections, author claims, contact requests, and site statistics; restricted by `PAPERLOG_ADMIN_EMAIL`.
- `/api/health` — database-backed health check.
- `/api/admin/export` — authenticated administrative JSON export for backup.
- `/privacy`, `/terms`, `/guidelines`, `/copyright`, `/contact` — launch and policy pages.

## Architecture

- Next.js 16 and React 19 application, compiled through vinext and Vite for a Cloudflare Workers-compatible runtime.
- Server-rendered pages and route handlers live under `app/`.
- D1 access and schema initialization live under `db/`.
- Drizzle schema and migrations live under `db/schema.ts` and `drizzle/`.
- OpenAlex integration lives primarily under `lib/openalex.ts`, `lib/openalex-browser.ts`, and the search/import routes.
- Sites deployment metadata is in `.openai/hosting.json`; it declares the `DB` D1 binding.
- Managed ChatGPT sign-in is isolated in `app/chatgpt-auth.ts`. This is the principal hosting-specific component to replace if the application moves away from Sites.

## Environment and secrets

The repository must never contain secret values. The expected variables are documented in `.env.example`:

- `OPENALEX_API_KEY` — optional but recommended for normal OpenAlex capacity.
- `PAPERLOG_ADMIN_EMAIL` — identifies the operator allowed to use administrative routes.

Before every public push, confirm that `.env*`, `.wrangler/`, `dist/`, `.vinext/`, `node_modules/`, exported user data, and downloaded backups are untracked. Do not paste credentials into issues, pull requests, screenshots, or documentation.

## Data and operational safety

- User emails, profiles, comments, ratings, reports, and account activity are personal data and must be handled accordingly.
- Use `/api/admin/export` to produce a manual database backup before schema changes, hosting migrations, or significant public testing.
- Store backups outside the repository in a private, encrypted location.
- Test account export and deletion after any schema or identity change.
- Keep the privacy notice aligned with the fields actually collected and the hosting providers actually used.
- User-submitted content requires moderation and a functioning report/contact route.
- Do not collect payment-card data, protected health information, or unnecessary sensitive attributes.

## Security state and known gaps

The alpha has authenticated writes, parameterized database queries, React output escaping, input limits, application-level rate limits, reports, moderation, and account controls. The live site uses HTTPS.

Before wider public launch, continue the following work:

- Add an explicit Content Security Policy, HSTS, frame-ancestor protection, MIME-sniffing protection, referrer policy, and permissions policy where the hosting platform permits them.
- Add explicit same-origin or CSRF defenses for state-changing requests.
- Strengthen rate limiting so it cannot be bypassed by changing a submitted email or another user-controlled field.
- Update dependencies and clear the currently known moderate nested PostCSS advisories after verifying compatibility.
- Add abuse monitoring, backup automation, restore testing, and a documented incident-response path.
- Obtain an appropriate privacy/legal review before broad international promotion.

## Hosting decision

OpenAI Sites is valid for the current alpha, but it is in public beta. The recommended more established free-path experiment is Cloudflare Workers + D1.

The current application is already close to that runtime. A safe migration should be performed in parallel:

1. Keep the Sites production deployment unchanged.
2. Export the production database.
3. Create a separate Cloudflare Worker and D1 database.
4. Replace managed ChatGPT sign-in with a new identity layer.
5. For the first five private testers, Cloudflare Access with an email allowlist is the smallest option.
6. Deploy to a temporary `workers.dev` URL, import data, and run the complete tester script.
7. Test through a staging subdomain before changing the apex-domain DNS.
8. Move `paperlog.net` only after the new deployment passes acceptance checks and a rollback path is ready.

The Workers free tier has hard request, CPU, and D1 daily limits and no free-plan uptime SLA. A migration is therefore an experiment to verify with actual traffic, not an assumption that all constraints disappear.

## Recovery and ownership

- Domain ownership is independent of the host. Do not transfer the domain merely to change hosting.
- GitHub contains the recoverable application source and migration history.
- Database exports contain user-generated state and require separate private storage.
- If the public site misbehaves, restrict its audience or revert to a previously verified deployment before making further changes.
- Never delete the current Site or change production DNS until the replacement and the rollback have both been tested.
